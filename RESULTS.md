# Результаты и подход
- Решение построено как гибрид с LLM и локальной логикой (вся обработка которая выполняется без LLM). Простые случаи обрабатываются локально, LLM же используется только там, где помогает понять свободный текст. При возникновении ошибок с LLM проект продолжает работать на локальной логике.
- `extract()` сначала просит LLM вернуть пять полей в JSON: сумму, дату, ИНН, контрагента и предмет документа. Затем значения проверяются и нормализуются локально, поэтому формат результата не зависит от формулировки ответа модели.
- Локальное извлечение (без LLM) покрывает типовые подписи, товарные таблицы, договорные формулировки и частые OCR-подмены в числах; на текущем наборе оно определяет 32 из 35 полей.
- При проверке извлечения с LLM корректно определены 33 из 35 полей.
- `classify()` работает без LLM: взвешенные текстовые признаки определяют тип документа, близкие результаты превращаются в `unknown`. Порог `MIN_MARGIN = 1.0` выбран как компромисс между числом ручных проверок и риском уверенной ошибки;
- `check_subject()` решает очевидные PASS/FAIL локально и обращается к LLM только для неоднозначных расходов. Все 14 предметов определены правильно (без EDGE).
- Главные ограничения — суммы, записанные полностью словами, и сильно повреждённый OCR-текст без устойчивых подписей полей.

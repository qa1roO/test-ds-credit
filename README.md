## 1. Как запустить проект

```bash
python -m pip install -e .
python -m pytest
python -m evaluation --report all
```

Для запуска с LLM переименовать .env.example в .env и вставить туда `finegrained` HF-токен (и в правах выбрать `Make calls to Inference Providers`)

## 2. Использованные технологии

- **Python** — основная логика, регулярные выражения и нормализация.
- **Hugging Face Inference API** — извлечение полей из документов
  свободного формата и обработка неоднозначных предметов оплаты.
- **python-dotenv** — загрузка токена и настроек из локального `.env`.
- **pytest** — детерминированные unit-тесты без реальных сетевых запросов.
- **tabulate** — формирование читаемых Markdown-таблиц в терминале.

LLM выбрана как основной способ извлечения, поскольку структура и формулировки
финансовых документов могут отличаться. Локальные правила используются для
нормализации, простых случаев и работы без API.

## 3. Архитектура решения

```text
dataset/              тестовые документы и subjects_test.txt
assets/               графики и картинки для сравнения качества
src/
├── extraction.py     извлечение и нормализация полей
├── classification.py классификация документов
├── subject_check.py  проверка предмета оплаты
├── llm_client.py     Hugging Face, rate limit и обработка ошибок
└── evaluation.py     запуск пайплайна и формирование отчётов
tests/                unit-тесты с mock-ответами LLM
```

Пайплайн:

```text
текст документа
      ├─ extract() ─────── amount, date, inn, contractor, subject
      ├─ classify() ────── contract / spec / invoice / act / unknown
      └─ check_subject() ─ допустимость, confidence и объяснение
                              └─ evaluation → итоговый отчёт
```

`extract()` сначала запрашивает у языковой модели JSON с пятью полями, затем
нормализует сумму, дату и ИНН. При отсутствии токена, ошибке провайдера или
некорректном JSON используется локальная логика для явно подписанных полей.

`classify()` работает локально: ключевые признаки добавляют баллы каждому типу
документа. Если разрыв между двумя лучшими классами меньше `MIN_MARGIN = 1.0`,
возвращается `unknown`.

`check_subject()` обрабатывает очевидные разрешённые и запрещённые расходы по
ключевым словам. LLM вызывается только для неоднозначных формулировок, например
доставки, аренды или консультационных услуг.

## 4. Сделанные компромиссы

- Локальная логика довольно проста и не подгоняется под каждый документ
  датасета. Это уменьшает объём правил, но снижает качество без LLM.
- Если поле не найдено или ответ модели нельзя разобрать, возвращается `None`.
  Пропуск считается безопаснее выдуманного значения.
- Отдельная коррекция повреждённого OCR не реализована, поэтому возможны ошибки
  в названиях организаций и реквизитах.
- Классификация основана на прозрачных ключевых признаках, а не на отдельной
  ML-модели. Это упрощает решение, но хуже работает на документах без
  характерного заголовка.
- `MIN_MARGIN = 1.0` выбран как компромисс: меньший порог уменьшает число
  `unknown`, но повышает риск ошибки; больший делает классификатор осторожнее.
- Точность извлечения считается как доля совпавших полей. Все пять полей имеют
  одинаковый вес; текстовые значения сравниваются после нормализации.
- `confidence` является эвристикой, а не калиброванной вероятностью.

## 5. Как проверить работу сервиса

```bash
python -m pytest
python -m evaluation --report all
```

`pytest` проверяет:

- обязательные примеры из ТЗ;
- нормализацию сумм, дат и ИНН;
- классификацию документов;
- локальную и LLM-ветки `check_subject()`;
- обработку ошибок Hugging Face;
- 10-секундный rate limit через mocks;
- формат итоговых отчётов.

Полный отчёт содержит четыре части:

1. обязательные примеры из ТЗ;
2. извлечение пяти полей из всех документов;
3. классификацию документов;
4. PASS/FAIL/EDGE для предметов оплаты.

Валидация extraction показывает совпадения по каждому полю и отдельную таблицу
ошибок. На текущем датасете получено `33/35` корректных полей с LLM и `22/35`
на локальной логике. Классификация определила `7/7` документов.
